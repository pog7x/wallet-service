# Метрики планировщика Go (Goroutine Scheduler)

| Параметр                  | Значение              | Описание                                                                                                                                                                                                                                                                                                                                                                              |
| :------------------------ | :-------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **gomaxprocs**            | `8`                   | Количество логических процессоров (P), настроенных для программы. Значение определяется настройкой `GOMAXPROCS` (по умолчанию берутся из системы).                                                                                                                                                                                                                                    |
| **idleprocs**             | `7`                   | Количество простаивающих логических процессоров. idleprocs=7 при gomaxprocs=8 означает, что семь P простаивают, а работу ведёт один. `idleprocs=0` означало бы, что каждый P привязан к потоку (M), но не обязательно выполняет полезный код. P считается не-простаивающим, когда он привязан к какому-то M (который может крутиться в поиске работы).                                |
| **threads**               | `6`                   | Общее количество потоков операционной системы (M), используемых рантаймом Go в данный момент.                                                                                                                                                                                                                                                                                         |
| **spinningthreads**       | `0`                   | Количество потоков в состоянии _spinning_ (крутятся в цикле в поиске работы). Это компромисс: несколько потоков держатся «горячими» вместо полного усыпления, чтобы быстрее подхватывать новые готовые горутины без задержки на пробуждение. Число таких потоков ограничено половиной `gomaxprocs`.                                                                                   |
| **idlethreads**           | `3`                   | Количество простаивающих потоков, у которых нет работы.                                                                                                                                                                                                                                                                                                                               |
| **runqueue**              | `0`                   | Количество горутин, ожидающих в **глобальной** очереди планировщика. Большое значение означает, что горутины поступают быстрее, чем P успевают их разбирать, либо локальные очереди переполнялись и сбрасывали половину содержимого в глобальную. Горутины round-robin и TransferBatch большую часть времени заблокированы на каналах в `_Gwaiting`, а не стоят в очереди готовности. |
| **Локальные очереди (P)** | `[ 0 0 0 0 0 0 0 0 ]` | Массив, показывающий количество горутин в локальной очереди каждого логического процессора (P). В примере у каждого из восьми P в очереди по `0` горутин.                                                                                                                                                                                                                             |

# Анализ состояния планировщика Go (Scheduler Trace)

В данном документе представлен детальный разбор внутреннего состояния компонентов планировщика Go (`P`, `M`, `G`) на основе снимка, сделанного во время выполнения тестовой нагрузки. Материал структурирован по трём основным сущностям рантайма.

## 1. Структура P (Processor)

`P` — это контекст планировщика, необходимый для выполнения Go-кода. Количество `P` задаётся параметром `GOMAXPROCS`.

**Поля и их значения:**

- **`status`** — состояние `P` (число):
  - `0` = `_Pidle` (свободен, не привязан к потоку).
  - `1` = `_Prunning` (привязан к `M` и исполняет код Go).
  - `2` = `_Psyscall` (владелец ушёл в системный вызов).
  - `3` = `_Pgcstop` (остановлен для сборки мусора).
  - `4` = `_Pdead` (выключен, например, после уменьшения `GOMAXPROCS`).

- **`schedtick`** — счётчик срабатываний планировщика. Увеличивается каждый раз, когда `P` выбирает новую горутину. Отражает активность планирования.

- **`syscalltick`** — счётчик входов в системные вызовы. Растёт, когда горутина на этом `P` уходит в системный вызов. (В снимке у `P0` он равен `32`, что указывает на обслуживание ввода-вывода).

- **`m`** — идентификатор `M`, который сейчас владеет этим `P`. Значение `nil` означает, что `P` свободен (согласуется со `status=0`).

- **`runqsize`** — количество горутин в **локальной очереди** этого `P`. Ноль означает отсутствие накопленной готовой работы.

- **`gfreecnt`** — число освобождённых структур `G` (`_Gdead`), закешированных на этом `P`. Переиспользование дешевле нового выделения.

- **`timerslen`** — число таймеров, зарегистрированных на этом `P`. Каждый `P` держит свою кучу таймеров для избежания глобальной блокировки.

---

## 2. Структура M (Machine / OS Thread)

`M` — это системный поток ОС, который исполняет код.

**Поля и их значения:**

- **`p`** — идентификатор `P`, которым владеет данный `M`. Значение `nil` означает, что поток не держит `P` и не может исполнять Go-код.

- **`curg`** — идентификатор горутины, исполняемой на этом `M` прямо сейчас. `nil` — поток не исполняет горутину.

- **`mallocing`** — ненулевое значение во время выделения памяти. Защитный флаг, запрещающий прерывания.

- **`throwing`** — ненулевое значение во время аварийного завершения (`runtime.throw`). В штатной работе — ноль.

- **`preemptoff`** — строка с причиной временного запрета вытеснения. Пустое значение — вытеснение разрешено.

- **`locks`** — число внутренних блокировок. Пока значение ненулевое, `M` находится в критической секции и не может быть вытеснен. (В снимке у `M0` и `M2` значение `2` из-за остановки мира для трассировки).

- **`dying`** — ненулевое значение во время паники или завершения потока.

- **`spinning`** — находится ли поток в состоянии кручения (активного поиска работы). Это оптимизация для быстрого подхвата готовых горутин.

- **`blocked`** — заблокирован ли `M`. (В снимке у `M3`, `M4`, `M5` стоит `true` — это лишние запаркованные потоки).

- **`lockedg`** — идентификатор горутины, жёстко привязанной к этому `M` (через `runtime.LockOSThread`). (В снимке: `M0` привязан к `G1`, `M1` — к `G17`).

## 3. Структура G (Goroutine)

`G` представляет собой легковесную горутину.

**Поля и их значения:**

- **`status`** — состояние горутины (число). **Скобка с причиной ожидания может содержать устаревшие данные!** Ориентироваться нужно только на число.
  - `0` = `_Gidle`
  - `1` = `_Grunnable` (готова, ждёт `P`)
  - `2` = `_Grunning` (исполняется)
  - `3` = `_Gsyscall` (в системном вызове)
  - `4` = `_Gwaiting` (заблокирована)
  - `6` = `_Gdead` (завершена)
  - `9` = `_Gpreempted` (вытеснена)

  > **Важно:** Причина в скобках не всегда очищается. Например, статус `2(chan receive)` означает, что горутина **исполняется**, а строка `chan receive` осталась от прошлой блокировки.

- **`m`** — идентификатор `M`, на котором находится горутина. `nil` — не привязана к потоку.

- **`lockedm`** — идентификатор `M`, к которому горутина жёстко привязана. **Важно:** `0` здесь — это `M0`, а не отсутствие привязки (отсутствие обозначается `nil`).

## 4. Текущая картина (Анализ снимка)

В момент снимка зафиксирована следующая ситуация:

### Общие параметры

- **`GOMAXPROCS=8`**, но активен только **один** `P`.
- **`P0`** в состоянии `1` (`_Prunning`) под управлением `M0`.
- **`P1`–`P7`** в состоянии `0` (`_Pidle`) без потоков.
- Все локальные очереди пусты (`runqsize=0`).

### Активные сущности

- **`G1`** — главная горутина, исполняется на `M0`, привязана к `M0` (`lockedm=0`).
- Причина в статусе (`chan receive`) остаточная — горутина уже получила управление.

### Пассивные сущности (Артефакты)

- **`M3`, `M4`, `M5`** — запаркованные лишние потоки (без `P` и без горутины). Иллюстрируют, что число потоков может превышать число активных `P`.
- **`G5`** — служебная горутина **`GOMAXPROCS updater (idle)`**. Это артефакт Go 1.25, который периодически перечитывает лимиты cgroup и корректирует `GOMAXPROCS`.

### Особенности нагрузки

- Основные горутины (Round-Robin, TransferBatch) заблокированы на каналах (`_Gwaiting`), поэтому не занимают очереди готовности. Отсюда — пустые `runqsize`.

---

## 5. Ключевые выводы

1.  **Приоритет числа над строкой:** При анализе статуса горутины (`status`) необходимо игнорировать текстовую причину в скобках, так как она может быть устаревшей.
2.  **Привязка потоков:** Значение `lockedm=0` у горутины означает привязку к `M0` (а не её отсутствие). Это подтверждается парным значением `lockedg=1` у `M0`.
3.  **Лишние потоки:** Наличие `M` с `blocked=true` и `p=nil` показывает, что рантайм может создавать потоки под пиковые нагрузки, которые затем засыпают в ожидании работы.
4.  **Активность:** Несмотря на `GOMAXPROCS=8`, полезную работу вёдет только один поток. Это характерно для тестов с интенсивным ожиданием на каналах, а не с вычислительной нагрузкой.
